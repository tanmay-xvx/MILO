# MILO Full-Project Audit — July 2026

**Scope:** every claim in the project guide (`docs/milo-guide.html`) verified against the
actual codebase, plus a correctness pass over the receiver (Rust), host (Python), prompts,
CI, and docs. Every finding below was either fixed in this branch or explicitly deferred
with a reason.

**Method:** read `receiver/src/*` (lib, main, engine, hal, targets, transport),
`host/*` (cli, core, devices, flash, mcp, hil), `prompts/system_prompt.txt`, CI workflows,
and all docs; cross-checked each guide/README statement against the code; ran the full
test matrix (see Verification).

---

## 1. Findings and status

Severity: 🔴 broken behavior · 🟠 incorrect/unsafe behavior in some path · 🟡 doc/consistency drift

### Code — fixed in this branch

| # | Sev | Where | Finding | Fix |
|---|-----|-------|---------|-----|
| C1 | 🔴 | `receiver/Cargo.toml` + `.cargo/config.toml` | **ESP32-C3 firmware build broken — CI red on `main` since May.** `portable-atomic` was given the `critical-section` feature globally while esp-hal enables `unsafe-assume-single-core` (and the rustflags cfg duplicated it); portable-atomic hard-errors on the combination. Every `Build MILO Receiver` run failed. | Removed the global `critical-section` feature and the redundant rustflags cfg; the `rp2040` feature now adds `portable-atomic/critical-section` (impl provided by rp2040-hal). Both targets compile. |
| C2 | 🔴 | `host/core/compiler.py` | **`get_param` missing from the extern template** — the 12th syscall was registered on-device and whitelisted, but LLM-generated code could not call it (compile error). Same omission in `prompts/system_prompt.txt`. | Added `fn get_param(slot: u32) -> u32;` to `RUST_LIB_TEMPLATE` and documented it in the system prompt. |
| C3 | 🔴 | `host/cli.py` | **`--subprocess` mode documented (README + guide) but did not exist** — laptop-only testing through the CLI was impossible; the flag was lost in the subcommand refactor. | Added `SubprocessTransport` (stdin/stdout framing with timeouts) to `core/transport.py` and a `--subprocess CMD` flag to `run`; autoflash is skipped in subprocess mode. Verified end-to-end. |
| C4 | 🟠 | `receiver/src/main.rs` (Pico + std) | **Wasm import validation only ran on the ESP32 `main_loop`.** The Pico USB loop and the std `--stdin` loop executed unvalidated modules, contradicting the documented security model. | Both loops now call `validate_wasm_imports` and reject non-Alphabet imports with the same error JSON as ESP32. |
| C5 | 🟠 | `receiver/src/lib.rs`, `main.rs` | **Execution-result JSON built without escaping** — a log line containing `"` or `\` produced invalid JSON and a host-side parse error. The `rejected imports: {:?}` messages were *always* invalid JSON (Debug-formatting a `Vec<String>` embeds quotes). | Added `json_escape()`; `exec_result_to_json` (now `pub`, reused by the Pico loop instead of a hand-rolled duplicate) escapes logs and errors; rejected-import messages are comma-joined and escaped. |
| C6 | 🟠 | `receiver/src/main.rs` (ESP32) | **Manifest advertised 14-bit PWM but the LEDC timer is configured as `Duty10Bit`** — the LLM was told a resolution the hardware isn't running. | Manifest now says `resolution_bits: 10`, matching the timer config and README. |
| C7 | 🟠 | `receiver/src/main.rs` (std) | **std receiver ran with *no* fuel by default** while its own manifest advertises `fuel_default: 10_000_000` — an infinite-loop driver hung the laptop receiver forever. | Default fuel is now `Some(10_000_000)` (matches manifest), still overridable via `--fuel`. |
| C8 | 🟡 | `receiver/src/engine/manifest.rs` | **`ALPHABET` constant had 11 entries** (missing `get_param`) despite the "keep in sync with register_syscalls" comment; discovery manifests under-reported the syscall set. | Added `get_param`; test now asserts 12 entries including it. |
| C9 | 🟡 | `host/cli.py` | `_maybe_autoflash` docstring said it runs `milo init --yes` but it passes `yes=False` (interactive confirm). | Docstring corrected; interactive confirm kept deliberately (flashing should not be silent). |

### Verified accurate (no change needed)

- The **12-syscall table** in guide/README: names, signatures, and semantics all match
  `register_syscalls` in `lib.rs` (after C2/C8, every layer agrees on 12).
- **MILO-Link framing** `[opcode u8][len u32 BE][payload]` and opcodes `0x01–0x09`:
  identical constants in `engine/link.rs` and `host/core/transport.py`.
- **Wasm limits**: 64 KB memory / 8 KB stack enforced by linker flags in `compiler.py`;
  matches manifests and prompts.
- **Fuel metering** (500M on-device), retry count (`MAX_COMPILE_RETRIES = 2`), GPT-4o model,
- ESP32 pins (LED/PWM GPIO 5, ADC ch 0 = GPIO 2, I2C SDA 8/SCL 9) and Pico pins
  (LED 25, ADC 26, I2C0 SDA 4/SCL 5) — all match `main.rs` and the target HALs.
- Known-gaps list (Wi-Fi stub, Pico single-core, sequential `push_to_all`, no on-device
  peripheral whitelist) — all confirmed accurate against the code.
- License: Apache-2.0 file present; README/guide statements correct.

## 3. Deferred / recommended follow-ups

2. **Rewrite `docs/PROJECT_WALKTHROUGH.md`** post-rename (banner added as a stopgap).

   provider switch (nice with MCP story) or drop the dependency.
