How it works

One sentence in, running machine code out

The host hands your request to the model together with the device’s own hardware manifest — the pins, buses and limits it reported at discovery. The generated Rust is wrapped in a no_std crate, compiled to a memory‑capped wasm module, and streamed to the device over MILO‑Link, a five‑byte‑header protocol that runs over anything that moves bytes.

Before the first instruction executes, the receiver has checked the frame size, verified an Ed25519 signature if the fleet requires one, and rejected any import that isn’t on the whitelist. Then it runs the module under fuel metering, inside a sandbox that cannot reach firmware memory. The model is creative; the runtime is paranoid.

The manifest is the source of truth. If a pin isn’t listed, the model is told so — and if generated code tries anyway, the attempt is denied and logged, never silently ignored.
gpio_set·gpio_get·delay_ms·get_uptime_us·i2c_transfer·spi_transfer·uart_write·uart_read·pwm_set·adc_read·log_msg·get_param

The Atomic Alphabet — every driver the model writes is composed of these twelve calls and nothing else. No malloc, no printf, no operating system.

Scenario one · the swarm

A laptop flies five drones, retasks them mid‑air, and repairs one in flight

Five emulated quadcopters run one 943‑byte controller that reads its altitude target from a live parameter slot — the echelon formation is data, not code. Eleven seconds in, the whole formation is retasked to a new altitude with a broadcast that takes five hundredths of a millisecond. No recompile. No reflash.

Then a motor fault is injected, and the interesting part begins. A proportional controller under a thrust deficit doesn’t crash; it settles 1.9 metres below where it was told to be, exactly as control theory predicts. The host asks GPT‑4o for a controller with integral action.6.9 seconds later a 975‑byte module is hot‑swapped onto the running drone; it clears the import whitelist, boots inside the sandbox, and flies the drone back into formation in 9.5 seconds. The other four never notice.

Every number on this page was measured on the run committed to the repository — transcripts, telemetry and the model’s verbatim source live in demos/evidence/. Two commands reproduce it.

The run, as the operator saw it. Total elapsed: 48 seconds, five clean landings.

Scenario two · the factory cell

Three machines, two unannounced failures, zero humans

A conveyor, a reflow oven running naive open‑loop vendor firmware, and a pick arm, all under one control station. When the oven’s heater element drifts 35% hot, a new set‑point wouldn’t help — the firmware itself is wrong. The station catches 175.9 °C against a 165 °C limit, GPT‑4o diagnoses the drift from cycle telemetry and writes a closed‑loop controller, 835 bytes in 4.7 seconds. Pushed between production cycles, it pulls the oven back inside limits within one cycle and holds 148 °C for the rest of the shift — on hardware whose behaviour had changed underneath it.

Later, an item jams the belt. Cycle ten reports the motor at commanded duty and zero items past the photo‑sensor; the supervisor halts the belt, sends the arm on a clear‑sweep confirmed by its limit switches, and production resumes. 4.1 seconds, detection to recovery — throughput identical to the healthy baseline.

The oven story is the one worth retelling: the right fix wasn’t a parameter, it was a new control law — written by the model, verified by the runtime, deployed without stopping the line.

The control plane

Any agent gets hands

MILO ships a complete Model Context Protocol server. Point Claude Code, Claude Desktop or Cursor at the repo and the agent can stand up an emulated fleet (milo_spawn_sim), register real boards over USB, read every device’s manifest, write firmware to them in natural language, retask them live, and hot‑swap repairs — eleven typed tools, all passing through the same four safety gates as everything else.

The trust model

Nothing the model emits is trusted

1

The frame is bounded

Payload lengths are checked against a 512 KB ceiling before any buffer is allocated. A hostile four‑gigabyte length prefix is refused, not honoured.

2

The signature is checked

With a trusted Ed25519 key provisioned, only operator‑signed modules run — verified on RISC‑V and ARM, not just the host. Unsigned pushes can be refused outright.

3

The imports are whitelisted

The module may import the twelve syscalls and nothing else. An unknown import means the module is never instantiated at all.

4

The sandbox is metered

64 KB of memory, 8 KB of stack, and per‑instruction fuel. An infinite loop runs out of gas and returns a clean error. The worst case is a failed episode.

A note on what “emulated” means here, precisely.

Every device in these demonstrations is a separate milo-receiver process — the same Rust runtime, wasmi interpreter, syscall ABI, validation, signing and framing that ship to ESP32‑C3 and Raspberry Pi Pico hardware (both build in CI today, as a flashable .bin and .uf2). Only the peripheral layer is simulated: small physics models for thrust, heat and belt motion, so the model’s code faces consequences. The GPT‑4o calls are live and the generated firmware is committed verbatim. Reproduce all of it:

cd receiver && cargo build && cd .. python3 demos/swarm_demo.py # the swarm, ~50 s python3 demos/factory_demo.py # the factory, ~55 s
