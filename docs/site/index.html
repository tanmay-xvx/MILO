No toolchain rituals. No reflash cycles. Firmware becomes a conversation.

Pipeline · english → silicon

One sentence becomes running machine code, with a safety story at every hop

The host feeds your request and the device's self-reported hardware manifest to the LLM. The generated Rust is wrapped in a no_std crate, compiled to a memory-capped wasm module, and streamed over MILO-Link — nine opcodes on any byte stream. The receiver validates every import against a 12-call whitelist before a single instruction executes, then runs it under fuel metering in a 64 KB sandbox.

"hold 3 m altitude"→ GPT-4o→ Rust→ wasm · 64 KB→ MILO-Link→ device→ telemetry back

Scenario A · drone swarm · demos/swarm_demo.py

A laptop flies five drones, retasks them mid-flight, and repairs one — in 48 seconds

Five emulated quadcopters run one 943-byte controller that reads its altitude target from a live parameter slot: the echelon formation is data, not code. Mid-flight, the whole formation is retasked with a 0.05 ms broadcast. Then a motor fault is injected — and the interesting part begins.

The fault droops, the LLM ships the fix, the drone climbs back

A pure-proportional controller under a thrust deficit doesn't crash — it settles 1.9 m below its commanded altitude, exactly as control theory predicts. The host asks GPT-4o for a replacement controller with integral action; 6.9 seconds later a 975-byte module is hot-swapped onto the running drone. It passes the import whitelist, boots inside the sandbox, and flies the drone back to formation in 9.5 s. The other four never notice.

Scenario B · autonomous cell · demos/factory_demo.py

A factory cell that diagnoses itself — and rewrites its own firmware

Three machines — a conveyor, a reflow oven on naive open-loop "vendor firmware", and a pick arm — run under one control station. Two incidents are injected without telling the controller. Both are handled autonomously, with GPT-4o as the supervisor brain and MILO as its hands.

Incident A — the heater drifts, so the LLM replaces the control law

When the oven's element starts running 35% hot, a new set-point wouldn't fix it — the firmware itself is wrong. The station catches 175.9 °C against a 165 °C limit, GPT-4o diagnoses the drift from cycle telemetry, writes a closed-loop integral controller (835 bytes, 4.7 s), and the station pushes it between production cycles. The oven pulls back inside limits within one cycle and holds 148 °C for the rest of the shift — on hardware whose characteristics had changed under it.

Incident B — a jam, diagnosed and cleared by another machine

Cycle 10: belt powered, zero items past the photo-sensor. The supervisor halts the belt, dispatches the arm for a clear-sweep (limit switches confirm full travel), and restores production — 4.1 seconds from detection to recovery, throughput identical to the healthy baseline.

Trust model · why letting an LLM write firmware isn't reckless

The AI is creative. The runtime is paranoid.

Import whitelist

Every module's imports are scanned against the 12-syscall Alphabet before instantiation — on every entry point. A module asking for anything else is rejected with a named error, never executed.

64 KB sandbox

Generated code runs in wasm with a hard 64 KB memory / 8 KB stack cap. No pointers into the firmware, no buffer overflows into the radio stack. The worst a bad driver can do is fail its own episode.

Fuel metering

Every instruction is metered. An infinite loop exhausts its gas and returns a clean fuel exhausted error — demonstrated in the test suite, not promised.

Live control plane

Devices answer stop, status, parameter and hot-swap requests while a driver runs — the host is never locked out of a machine that's busy. Parameters are how a whole fleet retasks in microseconds.

What "emulated" means here — precisely. Every device in these demos is a separate milo-receiver process: the same Rust runtime, wasmi interpreter, syscall ABI, validation and framing that ship to ESP32-C3 and Raspberry Pi Pico hardware (both boards run today over USB). Only the peripheral layer is simulated — small physics models for thrust, heat and belt motion, so the LLM's code faces consequences. The GPT-4o calls are live; the generated firmware is committed verbatim under demos/evidence/. What emulation doesn't cover: sensor noise, link flakiness, MCU timing jitter. Reproduce everything:
