What Is MILO?

MILO is an ultra-lightweight hardware abstraction layer that turns microcontrollers into a programmable extension of an LLM's reasoning engine. You describe what you want in natural language, the LLM writes Rust firmware, it compiles to WebAssembly, and the device executes it—often in seconds.

It bridges three worlds:

  • Natural language: You ask for a feature in plain English.
  • Rust + WebAssembly: An LLM (like GPT-4o) generates firmware code compiled to a tiny, sandboxed wasm binary.
  • Embedded hardware: The wasm runs on the target device (ESP32-C3 or Raspberry Pi Pico) with direct access to GPIO, I2C, ADC, PWM, and more through a minimal syscall interface.

The result: a feedback loop where LLMs can prototype, test, and refine hardware interactions without a C compiler toolchain or low-level debugging on every iteration.

Why MILO exists

Traditional embedded development is slow. You write C, compile for the target, flash it over JTAG, hope it works, and debug over serial. MILO eliminates most of that friction by treating the device as a wasm runtime that can be updated over USB serial in seconds. The LLM becomes your firmware engineer, and the feedback loop collapses from hours to seconds.

How It Works

The Flow

You → LLM → Rust code → wasm → Device → Result

In more detail:

  1. You type a task: blink the onboard LED three times
  2. The host sends your prompt + device manifest to GPT-4o
  3. GPT-4o generates a Rust function that calls syscalls like gpio_set() and delay_ms()
  4. The host compiles the Rust to WebAssembly (memory-constrained to 64 KB, 8 KB stack)
  5. The wasm binary is pushed over USB serial (1.2 KB for a simple blink, typically)
  6. The device's wasm runtime (wasmi, sandboxed) executes it with fuel metering ("gas") to prevent infinite loops
  7. Results and logs stream back over the same serial port
  8. If compilation fails, the host feeds the compiler error back to the LLM and retries (up to 2 times); device-side execution errors are reported to you but not auto-retried

Key insight: Wasm as the assembly language

By compiling to wasm instead of raw machine code, MILO gains:

  • Portability: Same wasm binary runs on any MILO receiver (ESP32, Pico, future targets).
  • Safety: The wasm sandbox prevents memory corruption, buffer overflows, and wild pointer dereferences.
  • Gas metering: CPU execution can be bounded with fuel limits to prevent runaway code.
  • Easy transport: A 1–2 KB binary is trivial to push over serial.

Architecture

MILO has two main parts: a Host (Python, on your laptop) and a Receiver (Rust firmware, on the device).

Host (Python) Receiver (Rust — ESP32-C3 or RP2040) ┌──────────────────────┐ ┌──────────────────────────────────┐ │ Natural language │ │ MiloRuntime + board HAL │ │ → GPT-4o │ │ ├─ wasmi (sandboxed wasm) │ │ → Rust → wasm │ │ ├─ syscall bindings + fuel ("gas") │ → serial │ ◄────► │ ├─ GPIO, PWM, ADC, I2C, SPI, UART │ (+ Wi-Fi WIP) │ MILO │ └─ EmbeddedHalAdapter │ │ milo init / download │ Link │ Transport: USB serial, USB CDC │ │ milo flash │ frames │ │ └──────────────────────┘ └──────────────────────────────────┘

Host components

CLI (cli.py)
Entry point. Manages device discovery, serial port selection, subprocess mode (laptop-only testing), and the main LLM loop where you type tasks.
Compiler (compiler.py)
Orchestrates Rust→wasm compilation. Wraps the LLM's function body in a #![no_std] crate and builds it with cargo (target wasm32-unknown-unknown, opt-level = "z"), capturing compiler errors for the retry loop.
Transport (transport.py)
Abstraction over serial and TCP. Implements MILO-Link framing: sending wasm bytecode, reading responses, handling discovery handshakes.
Device Management (devices/)
USB discovery, board identification (VID/PID map), multi-device registry for parallel operations.
Flash/Init (flash/)
Auto-detect blank boards, download firmware from GitHub releases, and flash with espflash or picotool.

Receiver components

MiloRuntime (lib.rs)
Glues together wasmi, the linker, and the host state. Registers all syscalls, validates imports, executes wasm, captures logs.
Executor (engine/executor.rs)
State machine for wasm execution. Tracks idle/running/completed/stopped states; submits bytecode; polls for results.
Hardware Abstraction (hal/adapter.rs)
EmbeddedHalAdapter: generic wrapper over any board's GPIO, PWM, ADC, I2C. Syscalls call methods on this adapter.
Board Implementations (targets/)
Concrete Esp32C3Hal and Rp2040Hal implementing the hardware traits. Board-specific pin mapping, clocking, and peripheral setup.
Transport (transport/)
USB serial (ESP32) or USB CDC (Pico). Reads frames, dispatches opcodes, writes results back.

Main loop (receiver)

The receiver runs an event loop that:

  1. Waits for a discovery request (host says "who are you?")
  2. Responds with a JSON hardware manifest (GPIO pins, ADC channels, I2C buses, etc.)
  3. Waits for a bytecode push (the compiled wasm binary)
  4. Validates wasm imports (whitelist: only syscalls allowed)
  5. Executes it, capturing logs and gas usage
  6. Sends back a JSON result: success, logs, any trapped errors
  7. Repeats

On the ESP32-C3 this is the shared main_loop in lib.rs, which also handles the extended control opcodes (stop, query status, set parameters, hot-swap). The Pico currently runs its own reduced USB-CDC loop that dispatches only discovery and bytecode push — extended opcodes are not yet wired there (see Known Gaps).

The Atomic Alphabet: 12 Syscalls

Every wasm driver communicates with hardware through exactly these functions, imported from module env:

Syscall Signature Purpose
gpio_set (pin: u32, state: u32) Set GPIO pin HIGH (1) or LOW (0)
gpio_get (pin: u32) → u32 Read GPIO pin state
delay_ms (ms: u32) Blocking delay in milliseconds
get_uptime_us () → u64 Microseconds since boot
i2c_transfer (addr, tx_ptr, tx_len, rx_ptr, rx_len) → i32 I2C read/write
pwm_set (channel: u32, duty: u32) Set PWM duty (0–10000 = 0–100%)
adc_read (channel: u32) → u32 Read ADC (resolution in manifest)
spi_transfer (bus, tx_ptr, tx_len, rx_ptr, rx_len) → i32 SPI read/write
uart_write (bus: u32, ptr: u32, len: u32) → i32 Write to UART bus
uart_read (bus, ptr, len, timeout_ms) → i32 Read from UART bus
log_msg (ptr: u32, len: u32) Log a UTF-8 message
get_param (slot: u32) → u32 Read host-writable parameter slot (0–7)

That's it. No malloc, no printf, no OS calls. Just these 12 primitives. Everything else (I2C to a sensor, PWM to an LED, UART to a module) is built on top of them.

Example: blinking an LED

#[unsafe(no_mangle)]
pub extern "C" fn run_logic() {
    for _ in 0..3 {
        unsafe {
            gpio_set(25, 1);           // LED on
            delay_ms(500);
            gpio_set(25, 0);           // LED off
            delay_ms(500);
        }
    }
}

Example: reading an ADC and displaying on SSD1306 OLED

More complex, but still using the same primitives: initialize the OLED over I2C with a command sequence, set the cursor, send bitmap font glyphs for the digit, and update on a loop.

Key constraint: Wasm memory is 64 KB with an 8 KB stack. The host's system prompt teaches the LLM how to clear the SSD1306 efficiently (8 pages, 128 columns), render numbers without alloc, and avoid large stack allocations.

Supported Hardware

ESP32-C3

  • Processor: 32-bit RISC-V, no native atomics (hence the wasmi patches)
  • Transport: USB Serial JTAG
  • GPIO: Flexible I/O
  • PWM: LEDC peripheral (10-bit, configurable frequency)
  • ADC: 12-bit
  • I2C: Standard I2C @ up to 400 kHz
  • Reference build: External LED (330 Ω) on GPIO 5, potentiometer on ADC channel 0 (GPIO 2), SSD1306 OLED on I2C @ 0x3C (SDA GPIO 8, SCL GPIO 9)

Raspberry Pi Pico (RP2040)

  • Processor: Dual-core ARM Cortex-M0+ (today MILO uses single-core)
  • Transport: USB CDC (virtual serial port)
  • GPIO: 26 I/O pins
  • PWM: 16 channels
  • ADC: 12-bit, 4-channel
  • I2C: 2 buses with optional recovery
  • Reference build: Onboard LED on GPIO 25, potentiometer on GPIO 26 (exposed as ADC channel 26 in the manifest), SSD1306 OLED on I2C0 @ 0x3C (SDA GPIO 4, SCL GPIO 5)

Future boards

Adding a new board requires:

  • A Rust HAL crate (e.g., stm32h7xx-hal)
  • An implementation of the MiloHardware trait
  • A Cargo feature (e.g., --features stm32h7)
  • Update to the manifest builder

The receiver's transport layer and wasm runtime are board-agnostic; only the HAL changes.

MILO-Link Protocol

Binary framing over any byte stream (USB serial, TCP, stdin/stdout):

[opcode: u8][payload_len: u32 big-endian][payload: bytes]
Opcode Direction Purpose
0x01 Bidirectional Discovery — JSON hardware manifest
0x02 Host → Device Bytecode push — raw wasm binary
0x03 Device → Host Execution result — JSON (success, logs, errors)
0x04–0x09 (Extended) Streaming, stop, query status, set parameter, hot-swap

Example handshake

  1. Host sends discovery: [0x01][0x00 0x00 0x00 0x00] (opcode + empty payload)
  2. Device responds: [0x01][0x00 0x00 0x01 0x42][{ JSON manifest }]
  3. Host sends bytecode: [0x02][payload_len in BE][wasm bytes...]
  4. Device executes, responds with result: [0x03][len][{ "ok": true, "logs": [...] }]

Hardware manifest

A JSON document (sent during discovery) describing what the device has:

{
  "board": "esp32c3",
  "family": "esp32",
  "firmware_version": "0.1.0",
  "capabilities": {
    "gpio": { "pins": [5] },
    "pwm":  { "pins": [5], "resolution_bits": 10 },
    "adc":  { "pins": [0], "resolution_bits": 12, "vref_mv": 3300 },
    "i2c": [
      { "bus_id": 0, "sda_pin": 8, "scl_pin": 9, "devices_present": ["0x3c"] }
    ]
  },
  "memory": { "ram_kb": 400, "flash_kb": 4096 },
  "wasm_limits": { "max_memory_kb": 64, "max_stack_kb": 8, "fuel_default": 500000000 },
  "alphabet": ["gpio_set", "gpio_get", "delay_ms", "...", "get_param"]
}

The LLM's system prompt includes this manifest, so it can pick valid pins and verify the device actually has the peripheral before generating code.

Key files to understand the codebase

  • receiver/src/lib.rs: Start here. Contains main_loop, syscall registration, and the runtime loop.
  • receiver/src/engine/executor.rs: The state machine for wasm execution. How does a wasm binary get loaded and run?
  • receiver/src/engine/link.rs: Frame format, opcodes, serialization.
  • host/cli.py: The interactive LLM loop. How does the host communicate with the device and call the LLM?
  • host/core/transport.py: Frame I/O and serial/TCP abstraction.
  • prompts/system_prompt.txt: The system prompt fed to the LLM. It teaches GPT-4o about the syscalls, memory constraints, and OLED rendering.

Getting Started

Prerequisites

You need:

  • Rust: nightly + stable with targets wasm32-unknown-unknown, riscv32imc-unknown-none-elf (ESP32), thumbv6m-none-eabi (Pico)
  • Python 3.10+: with openai, pyserial, esptool
  • espflash or picotool for flashing
  • OpenAI API key: in OPENAI_API_KEY env var

Flash a device

Download a prebuilt firmware binary (no compilation needed):

cd host
python3 cli.py init --port /dev/cu.usbmodem3101 -y

Or build from source:

cd receiver
cargo +nightly build --release \
  --target riscv32imc-unknown-none-elf \
  --features esp32c3 --no-default-features

espflash flash --port /dev/cu.usbmodem3101 \
  target/riscv32imc-unknown-none-elf/release/milo-receiver

Run the host

cd host
export OPENAI_API_KEY="sk-..."
python3 cli.py

Then type a task:

you → blink the onboard LED three times
you → read the potentiometer and show the value on the OLED
you → fade the LED brightness up and down using PWM

Laptop-only testing

No hardware? Build and run the mock receiver:

cd receiver
cargo build
cargo run -- ../examples/test_drivers/blink_led/target/wasm32-unknown-unknown/release/blink_led.wasm

cd ../host
python3 cli.py --subprocess "../receiver/target/debug/milo-receiver --stdin"

Note: the example driver wasm files are build artifacts — run cargo build --release --target wasm32-unknown-unknown inside each examples/test_drivers/* crate first (the receiver's integration tests need them too). The std receiver defaults to 10M fuel; override with --fuel N.

Development Status

Phases completed

Phase Theme Status
1 Transport abstraction Done USB serial on both boards, pluggable transport trait
2 Raspberry Pi Pico port Done Full RP2040 support, same host, USB CDC
3 WiFi transport Partial Host TCP client + mDNS; receiver TCP stub (not production)
4 Extended protocol & dual-core Partial Extended opcodes on ESP32; Pico single-core; Core 1 spawn not wired
5 Multi-device, MCP, safety Partial Registry + MCP scaffold + import validation; parallel push not implemented

What's working today

  • ESP32-C3: Full syscall alphabet, gas metering, import validation, parameter slots, hot-swap
  • Raspberry Pi Pico: Blink, ADC read, I2C (OLED), GPIO input, PWM
  • LLM loop: Iterative prompt → compile → push → execute, with retry on failure
  • Device discovery: USB VID/PID identification, board registry
  • Firmware provisioning: Download and flash from GitHub releases

Known gaps

  • WiFi: TCP transport on receiver is stubbed; no production mDNS or WiFi initialization
  • Dual-core (Pico): Core 1 is not used; both transport and Wasm run on Core 0
  • Parallel multi-device push: Registry supports push_to_all but it's sequential, not asyncio.gather
  • Extended opcodes (Pico): Discovery and bytecode push work; stop/query/set-param/hot-swap not dispatched on the USB loop
  • Peripheral whitelist: Manifest lists available pins, but the device doesn't reject out-of-spec syscalls; relies on HAL returning -1

On the horizon

The project's weekly docs (in docs/week3/) track a 5-phase roadmap. Future work includes finishing WiFi (real TCP + mDNS server on device), wiring Core 1 for parallel execution, and building an MCP agent that can orchestrate multi-device scenarios.
